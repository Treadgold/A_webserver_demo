The Absurd Extreme: A Minimal Web Server

  • Here's the opposite extreme – the world's simplest web server
  • Just 4 lines of Bash using netcat to listen on a port
  • No matter what you request – /, /about, /api/users – you get the same response
  • No file serving, no routing, no frameworks, no magic
  • But it WORKS! You could even point a domain at it and serve a page

#!/bin/bash
while true; do
  echo "HTTP/1.1 200 OK


Hello World
" | nc -l 8080
done

From the Absurd to Reality: Python Sockets

On the last slide, we saw the world’s simplest web server: one Bash script, one static response. Fun, but not real-world.

Now, we’ll start building a server the Python way, from the bottom up:

  • Set up a socket to listen for connections
  • Accept a client connection and send raw data
  • No frameworks yet—just the raw mechanics of networking
  • We’ll gradually layer features until it becomes a modern, async web server

Let’s jump into VS Code and see this in action, step by step.

import socket

def main() -> None:
    server_socket: socket.socket = socket.socket(
        socket.AF_INET,
        socket.SOCK_STREAM,
    )
    server_socket.bind(("127.0.0.1", 8002))
    server_socket.listen(1)

    print("Listening on http://127.0.0.1:8002")

    client_socket, addr = server_socket.accept()
    print(f"Connection from {addr}")

    # Send raw bytes — not valid HTTP
    client_socket.sendall(b"Hello from a socket!")
    client_socket.close()
    server_socket.close()


if __name__ == "__main__":
    main()

What this talk is

So let’s start very small

Before HTTP. Before frameworks.

Just a server, a socket, and a browser.

TCP is doing a lot for us

When we open a socket, we’re not starting from nothing.

TCP already guarantees ordering, delivery, retries, and flow control.

TCP makes sure:

  • Packets arrive in order
  • Lost packets are resent
  • Both sides agree the connection exists

This means our “simple” server is standing on a huge amount of machinery.

That’s not cheating. That’s the map.

Our first server

At the start, our server barely speaks HTTP.

It mostly just accepts a connection and doesn’t crash.

And that’s okay.

Because we know where we are on the map. We’re standing right on top of TCP.

Adding structure

Then we start recognizing patterns.

Requests. Headers. Status lines.

Nothing about the content changes.

The same HTML file. The same JavaScript. The same video.

Only the roads change

What changes is how we serve it.

Blocking vs non-blocking. One client vs many.

This is like upgrading from dirt roads to highways.

The destination stays the same.

Frameworks revisited

When we return to something like FastAPI, it’s no longer magic.

We can point to the map and say:

  • This part replaces our socket loop
  • This part handles concurrency
  • This part speaks HTTP for us

Ending where we started

The goal was never to memorize directions.

The goal was to know where you are.

If you leave with anything today, I hope it’s a better internal map.

One that lets you explore without fear of getting lost.
